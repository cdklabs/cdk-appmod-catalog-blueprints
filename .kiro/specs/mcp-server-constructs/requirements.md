# Requirements Document

## Introduction

This document defines the requirements for an MCP (Model Context Protocol) server that exposes the AppMod Catalog Blueprints library to AI-powered development tools. The MCP server enables AI models to scaffold, configure, and compose CDK L3 constructs from the `@cdklabs/cdk-appmod-catalog-blueprints` library by providing tools for code generation and resources for construct documentation. It is distributed as a standalone npm package, communicates over stdio transport, and uses JSII metadata to stay in sync with construct props automatically.

## Glossary

- **MCP_Server**: The Model Context Protocol server application that exposes tools and resources for scaffolding and configuring CDK constructs from the AppMod Catalog Blueprints library.
- **MCP_Tool**: An executable capability exposed by the MCP_Server that AI models invoke to generate CDK code snippets.
- **MCP_Resource**: A read-only data source exposed by the MCP_Server that AI models query to retrieve construct documentation, props interfaces, and architecture information.
- **Scaffold_Tool**: An MCP_Tool that generates CDK construct instantiation code with best-practice defaults and inline comments.
- **Compose_Tool**: An MCP_Tool that wires multiple constructs together in a single code snippet, handling cross-references between constructs.
- **Construct_Family**: A logical grouping of related constructs (e.g., Document Processing, Agents, Webapp, Foundation, Utilities).
- **JSII_Metadata**: Type information generated by the JSII compiler that describes construct props, types, and documentation. Used by the MCP_Server to auto-discover construct interfaces.
- **Props_Interface**: A TypeScript interface defining the configuration options for a CDK construct.
- **Code_Snippet**: A block of TypeScript code containing construct instantiation (the `new ConstructName(...)` block) that a user integrates into their own CDK stack.
- **Smart_Defaults**: Best-practice default values applied to optional construct props, with inline comments explaining each choice.
- **AI_Client**: An MCP-compatible AI tool (e.g., Claude Desktop, VS Code extension, Kiro) that connects to the MCP_Server over stdio.

## Requirements

### Requirement 1: Construct Scaffolding via Grouped Tools

**User Story:** As a developer using an AI assistant, I want to scaffold any CDK construct from the AppMod Catalog Blueprints library by invoking a capability-grouped MCP tool, so that I get working construct instantiation code without memorizing props interfaces.

#### Acceptance Criteria

1. THE MCP_Server SHALL expose Scaffold_Tools grouped by Construct_Family, where each Scaffold_Tool accepts a `constructType` parameter to select the specific construct within that family.
2. WHEN a developer invokes a Scaffold_Tool with a valid `constructType`, THE Scaffold_Tool SHALL return a Code_Snippet containing the construct instantiation with correct TypeScript imports and all required props populated.
3. WHEN a developer invokes a Scaffold_Tool without specifying optional props, THE Scaffold_Tool SHALL apply Smart_Defaults to all optional props and include inline comments explaining each default value.
4. WHEN a developer specifies override values for optional props, THE Scaffold_Tool SHALL use the provided values instead of Smart_Defaults in the generated Code_Snippet.
5. THE MCP_Server SHALL support scaffolding for all public constructs across every Construct_Family: Document Processing, Agents, Webapp, Foundation, and Utilities.

### Requirement 2: Construct Documentation via MCP Resources

**User Story:** As a developer using an AI assistant, I want to browse construct documentation, props interfaces, and architecture information through MCP resources, so that the AI model has full context before generating code.

#### Acceptance Criteria

1. THE MCP_Server SHALL expose MCP_Resources for each public construct, providing the construct's documentation, Props_Interface definition, and architectural description.
2. WHEN an AI_Client reads a construct MCP_Resource, THE MCP_Server SHALL return the construct's props with types, descriptions, default values, and required/optional status.
3. THE MCP_Server SHALL expose a catalog MCP_Resource that lists all available Construct_Families and the constructs within each family.
4. WHEN the library version changes, THE MCP_Server SHALL reflect updated construct documentation in MCP_Resources without requiring manual metadata updates, by reading from JSII_Metadata.

### Requirement 3: Multi-Construct Composition

**User Story:** As a developer, I want to compose multiple constructs into a single wired-up code snippet, so that I can quickly set up complex stacks where constructs reference each other (e.g., Network passed into BedrockDocumentProcessing).

#### Acceptance Criteria

1. THE MCP_Server SHALL expose a Compose_Tool that accepts a list of constructs and generates a single Code_Snippet wiring them together with correct cross-references.
2. WHEN a developer requests composition of constructs that depend on each other, THE Compose_Tool SHALL order construct instantiation so that dependencies are created before dependents (e.g., Network before BedrockDocumentProcessing).
3. WHEN a developer requests composition of constructs that share a common dependency, THE Compose_Tool SHALL create the shared dependency once and pass it to all constructs that require it.
4. IF a developer requests composition of constructs that are incompatible or have conflicting props, THEN THE Compose_Tool SHALL return a clear error message identifying the conflict and suggesting a resolution.

### Requirement 4: JSII Metadata Integration

**User Story:** As a maintainer of the MCP server, I want construct props and types to be auto-discovered from JSII metadata, so that the MCP server stays in sync with the library without manual metadata maintenance.

#### Acceptance Criteria

1. THE MCP_Server SHALL read Props_Interface definitions from JSII_Metadata generated by the library's build process.
2. WHEN a new construct is added to the library and JSII_Metadata is regenerated, THE MCP_Server SHALL automatically discover the new construct's props without code changes to the MCP_Server.
3. WHEN a construct's Props_Interface changes (props added, removed, or type-changed), THE MCP_Server SHALL reflect those changes in both Scaffold_Tools and MCP_Resources after the library is rebuilt.
4. THE MCP_Server version SHALL be locked to the library version, ensuring that the JSII_Metadata and MCP_Server capabilities are always consistent.

### Requirement 5: stdio Transport and Distribution

**User Story:** As a developer, I want to run the MCP server via `uvx` and connect it to my AI tool over stdio, so that setup is a single command with no infrastructure to manage.

#### Acceptance Criteria

1. THE MCP_Server SHALL communicate with AI_Clients exclusively over the stdio transport protocol.
2. THE MCP_Server SHALL be implemented in Python and published as a standalone PyPI package runnable via `uvx @cdklabs/mcp-appmod-catalog-blueprints`.
3. WHEN an AI_Client connects to the MCP_Server over stdio, THE MCP_Server SHALL complete the MCP initialization handshake and advertise all available tools and resources within 5 seconds on a standard development machine.
4. THE MCP_Server SHALL operate as a single-process application with no external runtime dependencies beyond Python (minimum version 3.10).

### Requirement 6: Error Handling and Graceful Degradation

**User Story:** As a developer, I want the MCP server to handle incomplete or invalid requests gracefully, so that I always get useful output even when my request is imprecise.

#### Acceptance Criteria

1. WHEN a developer invokes a Scaffold_Tool without specifying all required props, THE Scaffold_Tool SHALL generate a Code_Snippet with placeholder values for missing required props and include comments indicating what needs to be filled in.
2. WHEN a developer invokes a Scaffold_Tool with an invalid `constructType`, THE MCP_Server SHALL return an error message listing the valid construct types for that Construct_Family.
3. IF the JSII_Metadata file is missing or unreadable, THEN THE MCP_Server SHALL start in a degraded mode, return a clear error message on tool invocation explaining that metadata is unavailable, and suggest how to resolve the issue.
4. WHEN a developer provides a prop value with an incorrect type, THE Scaffold_Tool SHALL include the value as-is in the Code_Snippet with a warning comment indicating the expected type.

### Requirement 7: Smart Defaults and Code Quality

**User Story:** As a developer, I want generated code to follow the library's best practices and coding standards out of the box, so that I start with production-quality code rather than a bare skeleton.

#### Acceptance Criteria

1. THE Scaffold_Tool SHALL generate Code_Snippets that follow the repository's coding standards: 2-space indentation, single quotes, semicolons, and PascalCase construct IDs.
2. THE Scaffold_Tool SHALL apply security best practices in Smart_Defaults, including KMS encryption for S3 buckets, enforceSSL where applicable, and least-privilege IAM patterns.
3. THE Scaffold_Tool SHALL include JSDoc-style inline comments on each prop explaining its purpose and the rationale for the chosen default value.
4. WHEN generating Code_Snippets for constructs that support observability, THE Scaffold_Tool SHALL include observability configuration as a commented-out optional section with instructions for enabling it.

### Requirement 8: Construct Family Coverage

**User Story:** As a developer, I want the MCP server to cover all construct families in the library, so that I have a single tool for scaffolding any part of my infrastructure.

#### Acceptance Criteria

1. THE MCP_Server SHALL provide scaffolding support for the Document Processing family: `BedrockDocumentProcessing`, `AgenticDocumentProcessing`, adapters (`QueuedS3Adapter`), and chunking configuration.
2. THE MCP_Server SHALL provide scaffolding support for the Agents family: `BatchAgent`, `InteractiveAgent`, and knowledge base configuration.
3. THE MCP_Server SHALL provide scaffolding support for the Webapp family: `Frontend` construct for CloudFront and S3 hosting.
4. THE MCP_Server SHALL provide scaffolding support for the Foundation family: `Network` (VPC), `AccessLog`, and `EventBridgeBroker`.
5. THE MCP_Server SHALL provide scaffolding support for the Utilities family: `Observability`, `DataMasking`, `DataLoader`, and CDK Nag helpers.
6. WHEN a new construct is added to any Construct_Family and JSII_Metadata is regenerated, THE MCP_Server SHALL make the new construct available for scaffolding without MCP_Server code changes.
