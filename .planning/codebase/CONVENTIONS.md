# Coding Conventions

**Analysis Date:** 2026-03-01

## Naming Patterns

**Files:**
- Construct/class files use PascalCase: `access-log.ts`, `base-agent.ts`, `network.ts`
- Interface files prefixed with `i-`: `i-knowledge-base.ts`
- Props files suffixed with `-props`: `knowledge-base-props.ts`
- Test files suffixed with `.test.ts` or `.test.ts`: `access-log.test.ts`
- NAG security tests suffixed with `-nag.test.ts`: `framework-nag.test.ts`
- Index files use `index.ts` for barrel exports

**Functions:**
- Use camelCase for all functions and methods: `createServiceEndpoint()`, `generateIamPermissions()`, `exportConfiguration()`
- Static utility methods use camelCase: `BedrockModelUtils.deriveActualModelId()`, `LogGroupDataProtectionUtils.handleDefault()`
- Constructor named `constructor`

**Variables:**
- camelCase for all variables and properties: `bucketName`, `stackId`, `testModel`
- Private properties and methods prefixed with underscore: `private readonly props: NetworkProps`
- Public readonly properties declared with `readonly` keyword: `public readonly bucket: s3.Bucket`
- Constants that are configuration use camelCase: `baseName`, `defaultLifecycleRules`

**Types & Interfaces:**
- PascalCase for all interfaces: `AccessLogProps`, `NetworkProps`, `BedrockModelProps`, `IKnowledgeBase`
- Interfaces suffixed with `Props` for configuration objects: `AgentDefinitionProps`, `BaseKnowledgeBaseProps`
- Interfaces prefixed with `I` for abstract/implemented contracts: `IVpc`, `IKnowledgeBase`, `ILayerVersion`
- Enums use PascalCase with UPPERCASE values: `BedrockCrossRegionInferencePrefix` with values `US`, `EU`

## Code Style

**Formatting:**
- Configured by `.eslintrc.json` (generated by projen)
- Indentation: 2 spaces (enforced by `@stylistic/indent`)
- Line length: 150 characters maximum (enforced by `@stylistic/max-len`, ignores URLs, strings, templates, comments, regex)
- Quotes: Single quotes preferred with escape avoidance (`@stylistic/quotes`)
- Semicolons: Always required at end of statements (enforced by `@stylistic/semi`)

**Spacing:**
- Curly braces: Always use braces for blocks even if single line, 1TBS style (`@stylistic/brace-style`)
- Object property spacing: Always include spaces around braces: `{ property: value }` (enforced by `@stylistic/object-curly-spacing`)
- Array bracket spacing: No spaces inside brackets: `[item]` (enforced by `@stylistic/array-bracket-spacing`)
- Comma spacing: Space after comma, no space before (enforced by `@stylistic/comma-spacing`)
- Trailing commas: Always use multiline commas (enforced by `@stylistic/comma-dangle: "always-multiline"`)

**Example formatting:**
```typescript
// Correct
const obj = { key: 'value', another: 'test' };
const arr = [1, 2, 3];
const multiline = {
  prop1: 'value1',
  prop2: 'value2',
};
if (condition) {
  doSomething();
}
```

**Linting:**
- ESLint with TypeScript plugin enabled (`@typescript-eslint/eslint-plugin`, `@typescript-eslint/parser`)
- Import ordering: Imports grouped as [builtin, external] with alphabetic sorting (enforced by `import/order`)
- No extraneous dependencies (enforced by `import/no-extraneous-dependencies` with devDependencies allowed)
- Curly braces enforced with `curly: ["error", "multi-line", "consistent"]`
- No bitwise operators allowed (enforced by `no-bitwise`)

## Import Organization

**Order:**
1. Builtin/core modules: `import * as path from 'path'`
2. AWS CDK libraries: `import * as cdk from 'aws-cdk-lib'`, `import * as iam from 'aws-cdk-lib/aws-iam'`, etc.
3. Constructs library: `import { Construct } from 'constructs'`
4. Relative imports (utilities, local modules): `import { NetworkProps } from './network'`
5. Re-exports grouped at top after imports: `export { InvokeType }`

**Path Aliases:**
- Not used in this project; relative imports with explicit paths
- Imports always from specific modules, not from package roots unless using index re-exports

**Example import pattern:**
```typescript
// Copyright Amazon.com, Inc. or its affiliates. All Rights Reserved.
// SPDX-License-Identifier: Apache-2.0

import * as cdk from 'aws-cdk-lib';
import * as iam from 'aws-cdk-lib/aws-iam';
import * as s3 from 'aws-cdk-lib/aws-s3';
import { Construct } from 'constructs';
import { LambdaIamUtils } from '../../utilities';
import { Network } from '../foundation';
import { InvokeType } from './invoke-type';

export { InvokeType };
```

## Error Handling

**Patterns:**
- Explicit error throwing with descriptive messages: `throw new Error('name is required and cannot be empty')`
- Validation at construction time in constructor or `validateProps()` method
- Input validation before processing to catch errors early
- Error messages should describe what went wrong and how to fix it

**Example:**
```typescript
protected validateProps(props: BaseKnowledgeBaseProps): void {
  if (!props.name || props.name.trim() === '') {
    throw new Error('name is required and cannot be empty');
  }
  if (!props.description || props.description.trim() === '') {
    throw new Error('description is required and cannot be empty');
  }
}
```

## Logging

**Framework:** `console` (not abstracted)

**Patterns:**
- Not extensively used in codebase; rely on AWS CloudWatch and CDK synthesis logs
- When needed, use standard `console.log()`, `console.error()`, `console.warn()`
- For Lambda functions: use Lambda Powertools when observability is enabled
- AgentCore observability configured via environment variables for agent-specific logging

## Comments

**When to Comment:**
- All exported classes require JSDoc comments
- All public methods require JSDoc comments
- Public properties require inline JSDoc comments
- Complex internal logic should have inline comments
- Props interfaces documented with property descriptions
- All parameters and return types documented in JSDoc

**JSDoc/TSDoc:**
- Use `/**` style blocks for all public APIs
- Document parameters with `@param`: `@param serviceName The name of the service (e.g., 'alb', 'cloudfront', 's3')`
- Document return type with `@returns`: `@returns The S3 path for the service's access logs`
- Document defaults with `@default`: `@default false` or `@default 5`
- Document errors with `@throws`: `@throws Error if the region is not supported`
- Use `@see` for references to external documentation: `@see https://docs.aws.amazon.com/...`
- Example usage in JSDoc comments enclosed in code blocks with backticks

**Example JSDoc:**
```typescript
/**
 * Get the S3 bucket path for a specific service's access logs
 *
 * @param serviceName The name of the service (e.g., 'alb', 'cloudfront', 's3')
 * @param resourceName Optional resource name for further organization
 * @returns The S3 path for the service's access logs
 */
public getLogPath(serviceName: string, resourceName?: string): string {
  const basePath = `${this.bucketName}/${this.bucketPrefix}/${serviceName}`;
  return resourceName ? `${basePath}/${resourceName}` : basePath;
}
```

## Function Design

**Size:** Keep functions focused and under 100 lines when practical; many functions are 20-50 lines

**Parameters:**
- Use object destructuring for props: `constructor(scope: Construct, id: string, props: AccessLogProps = {})`
- Always include type annotations
- Use optional parameters with `?`: `resourceName?: string`
- Provide sensible defaults: `props: NetworkProps = {}`

**Return Values:**
- Always explicitly type return values: `public getLogUri(...): string`
- Return early to reduce nesting
- Use `readonly` for immutable return values

**Example:**
```typescript
constructor(scope: Construct, id: string, props: AccessLogProps = {}) {
  super(scope, id);
  this.bucketPrefix = props.bucketPrefix || 'access-logs';
  const accountId = cdk.Stack.of(this).account;
  const region = cdk.Stack.of(this).region;
  // ... rest of initialization
}
```

## Module Design

**Exports:**
- Use `export class`, `export interface`, `export enum` for public APIs
- Use barrel exports in `index.ts`: `export * from './file-name'`
- Convention: One main class/interface per file, supporting types in same file

**Barrel Files:**
- `index.ts` files use wildcard exports
- Location: `use-cases/framework/foundation/index.ts`, `use-cases/framework/agents/index.ts`
- Purpose: Aggregate related exports for cleaner imports
- Pattern: Consecutive `export * from './module-name'` statements

**Example barrel file:**
```typescript
export * from './network';
export * from './access-log';
export * from './eventbridge-broker';
```

## TypeScript Strict Mode

**Configuration:**
- `strict: true` enabled in `tsconfig.json`
- `noImplicitAny: true` - all types must be explicit
- `noImplicitThis: true` - `this` context must be explicit
- `noImplicitReturns: true` - all code paths must return
- `noUnusedLocals: true` - no unused variables
- `noUnusedParameters: true` - no unused parameters
- `strictNullChecks: true` - must handle null/undefined
- `strictPropertyInitialization: true` - properties must be initialized

**Pattern:** All properties are explicitly typed and initialized:
```typescript
public readonly bucket: s3.Bucket;
public readonly bucketName: string;
private readonly props: NetworkProps;
protected readonly logGroupDataProtection: LogGroupDataProtectionProps;
```

## Class Structure

**Member Ordering** (enforced by `@typescript-eslint/member-ordering`):
1. Public static fields
2. Public static methods
3. Protected static fields
4. Protected static methods
5. Private static fields
6. Private static methods
7. Public instance fields
8. Constructor
9. Public instance methods
10. Protected instance methods
11. Private instance methods

**Example:**
```typescript
export class AccessLog extends Construct {
  // Public properties first
  public readonly bucket: s3.Bucket;
  public readonly bucketName: string;

  // Constructor
  constructor(scope: Construct, id: string, props: AccessLogProps = {}) {
    // ...
  }

  // Public methods
  public getLogPath(serviceName: string, resourceName?: string): string {
    // ...
  }

  public getLogUri(serviceName: string, resourceName?: string): string {
    // ...
  }
}
```
